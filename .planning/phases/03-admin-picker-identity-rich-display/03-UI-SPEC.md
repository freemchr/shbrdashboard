# Phase 3 — UI Design Contract

## Scope & Anti-Scope

**In scope (this spec governs):**

- `components/ui/PrimeUserPicker.tsx` — net-new shared inline combobox (D-02). Used by 3 call sites: Dashboard Admins, Group member editor, New Group form.
- `app/admin/visibility-tab.tsx` — net-new file extracted from `app/admin/page.tsx` per D-20. Hosts the Refresh button + metadata strip (D-13), Dashboard Admins list, Groups list, Group member rows, Page Access section (unchanged).
- `app/admin/audit-tab.tsx` — net-new file extracted per D-20. Hosts the audit table with the D-15 three-step actor cascade and D-16 `prime_user_miss` row tooltip.
- Two-line row pattern (D-06) wherever a Prime user is rendered on these surfaces (picker results, picker chips, group member rows, Dashboard Admins rows, audit log actor cells).
- Manual-email fallback UI (D-12) when `/api/admin/prime-users` errors or returns empty.
- Stale/unresolved-email tooltip treatment (D-09).

**Out of scope (do NOT specify):**

- TopBar identity label — locked in Phase 2 UI-SPEC (`02-UI-SPEC.md`); this spec re-uses tokens but does not modify TopBar.
- Page Access section in the visibility tab — pre-existing UI, no changes from this milestone.
- `app/admin/page.tsx` tab strip / tab routing — extraction per D-20 preserves the existing tab-button styling unchanged.
- `ActionBadge`, `formatAEDT`, `exportCSV` — relocated by D-20 but visually unchanged. CSV header rename per D-17 is a string change, not a visual change.
- Server-side enforcement, audit-config audit trail, BOOTSTRAP-01 fallback removal — all v2.
- Avatars, admin badges (other than the existing `Miss` audit badge), status-active dots, division/region/role/trade chips beyond the `· {division}` secondary suffix.

## Typography

The visibility tab and audit tab inherit the global `text-sm` body baseline established in Phase 2. Phase 3 declares **3 sizes** and **2 weights** total — no new fonts, no italic, no underline, no letter-spacing changes.

| Role | Size | Weight | Line Height | Tailwind | Usage |
|------|------|--------|-------------|----------|-------|
| Primary identity (Line 1) | 14px | 400 | 1.5 | `text-sm text-gray-300` | Picker results, chips, group/admin rows, audit actor: `fullName` (or `email` fallback) |
| Secondary meta (Line 2) | 12px | 400 | 1.4 | `text-xs text-gray-500` | Picker results, group/admin rows, audit actor: `email` + optional `· {division}` |
| Section heading | 14px | 600 | 1.5 | `text-sm font-semibold text-white` | Section card titles (e.g. "Dashboard Admins", "Groups") — unchanged from existing |
| Helper / refresh metadata | 12px | 400 | 1.4 | `text-xs text-gray-500` | Refresh result strip, helper text, picker error copy |
| Button label (primary CTA) | 14px | 500 | 1.5 | `text-sm font-medium` | "Save Changes", "Refresh Prime Users", "Add Email" — unchanged from existing |

**Declared rules:**

- **Two weights only:** 400 regular and 600 semibold for section headings. Buttons use 500 medium — this is the existing project pattern (CTAs: `font-medium`); declared as a third weight for buttons only to match what is already on the page. Identity rows never use 600/700.
- **No bold names.** The primary identity line is `font-normal` — distinguished from secondary by color (`text-gray-300` vs `text-gray-500`) and size (14px vs 12px), not weight. Matches Phase 2's "no bold" rule.
- **Tabular nums NOT applied.** Prime names aren't numeric. The refresh-metadata duration ("1.2s") is short enough to not need tabular alignment.
- **`truncate` allowed but not required:**
  - Picker dropdown rows: `truncate` on Line 1 and Line 2 with row-level `min-w-0` to prevent overflow when the dropdown sits in a narrow column.
  - Chip text: `max-w-[200px] truncate` (matches Phase 2 TopBar cap) so a chip never expands past 200px.
  - Group/admin list rows + audit table: no truncate cap by default (full names expected to fit; `· {division}` will always be `null` in tenant per probe). If a row overflows on narrow viewports, allow horizontal scroll on the parent (audit table only — see Responsive section).
- **Division separator** is the middle-dot `·` (U+00B7), with a single-space buffer on each side (`text-gray-500` — same color as Line 2, no weight emphasis). Lifted from Phase 2 D-13 implementation pattern; locked here. Do NOT use em dash `—`, en dash `–`, vertical bar `|`, or comma.

**Font family:** inherits `font-sans` (Inter) — no override.

## `<PrimeUserPicker>` — Anatomy & Interaction Contract

The single most spec-heavy surface in this phase. Used identically by all 3 admin call sites (D-02). Spec is exhaustive so the planner has no design ambiguity.

### Visual Anatomy

```
┌─ wrapper (relative, w-full)
│  ┌─ chip row (flex flex-wrap gap-2 mb-2)
│  │   [chip] [chip] [chip] …
│  └─
│  ┌─ input (w-full, role=combobox, aria-expanded, aria-controls)
│  │   "Search Prime users by name, email, or division…"
│  └─
│  ┌─ dropdown (absolute, top-full, mt-1, z-30, max-h-72 overflow-auto)
│  │   role=listbox, only rendered when (open && filtered.length > 0)
│  │   ┌─ row (role=option) ────────────────────────
│  │   │   [User icon 14px] Chris Freeman
│  │   │                    chris.freeman@techgurus.com.au
│  │   └────────────────────────────────────────────
│  │   ┌─ row (role=option, aria-selected=true) ──── (highlighted via keyboard)
│  │   │   [User icon 14px] Jane Doe
│  │   │                    jane.doe@shbr.com.au
│  │   └────────────────────────────────────────────
│  └─
│  ┌─ empty-results state (when open && query && filtered.length === 0)
│  │   text-sm text-gray-500 px-3 py-2 (rendered INSIDE the dropdown surface)
│  │   "No matches. Press Enter to add as raw email." (only when D-12 fallback eligible)
│  └─
│  ┌─ refresh-metadata strip (only on the picker that lives at top of Dashboard Admins;
│  │   the per-tab Refresh button at top of Visibility tab is the canonical one — see D-13)
│  └─
└─
```

**z-index:** Dropdown uses `z-30`. Above admin section cards (which have no z-index) and above the existing toast (which uses no z-index either, sitting in normal flow). Below modals (none in this phase). Below the TopBar (which uses `z-40`/`sticky` — verified in `components/ui/TopBar.tsx`). Choose `z-30` to leave one stop below TopBar and one stop above other floating UI.

**Dropdown shadow:** `shadow-lg shadow-black/40` — same as existing admin dropdowns/menus on dark theme. Provides depth without competing with section cards.

**Dropdown positioning:** `absolute top-full left-0 right-0 mt-1` — anchored to input, full-width of input. Does NOT use a portal. Does NOT use Popper / Floating UI. Plain CSS positioning is sufficient because:

1. The picker always sits inside a section card with enough room below.
2. The dropdown max-height (288px) caps overflow.
3. The 3 call sites (Dashboard Admins, Group editor, New Group form) all live in vertically-stacked sections — none are inside scrollable parents.
4. If a future call site needs to escape an `overflow-hidden` parent, that's a v2 portal-isolation upgrade. Not this phase.

### Inactive User Treatment (Claude's Discretion → locked here)

Per probe (CONTEXT.md "carried-forward foundation"), the SHBR tenant has `status` populated. Decision: **include inactive users in picker results** with `(inactive)` appended to Line 1 in `text-gray-500`. Do NOT filter them out — admins may need to remove a leaver's stale group membership and need to see them.

- Line 1: `Chris Freeman (inactive)` where `(inactive)` is wrapped in a `text-gray-500` span and the rest stays `text-gray-300`.
- No row dim (no `opacity-50`), no badge, no separate sort. The single-word "(inactive)" is enough.
- Selected chips of inactive users render the same way (with `(inactive)` appended) so a saved admin still sees the warning.
- Filter logic still matches against `fullName` / `email` / `division` (status is not searchable).

### Keyboard Interaction Model (locked, testable)

| Key | When | Behavior |
|-----|------|----------|
| Type any printable character | Input focused | Filter the dropdown live (no debounce — list ≤ ~30 records). Open dropdown if closed. Reset `activeIndex` to 0. |
| Focus enters input (click or Tab) | Input not focused | Open dropdown (`open=true`). Show all `availableUsers - selectedEmails` if input is empty; show filtered if input has text. |
| Blur input (click outside or Tab away) | Input focused | Close dropdown after a short delay (use `onMouseDown` on dropdown rows to add before blur fires) — no need for focus-trap logic since list selection is via click or Enter. |
| ArrowDown | Dropdown open | `e.preventDefault()`. `activeIndex = Math.min(activeIndex + 1, filtered.length - 1)`. Scroll active row into view if needed (`scrollIntoView({ block: 'nearest' })`). |
| ArrowUp | Dropdown open | `e.preventDefault()`. `activeIndex = Math.max(activeIndex - 1, 0)`. Scroll active row into view. |
| Enter | Dropdown open AND `filtered[activeIndex]` exists | `e.preventDefault()`. Add `filtered[activeIndex].email` to `selectedEmails`. Clear input. Reset `activeIndex` to 0. Keep dropdown open (so admin can keep typing). |
| Enter | Dropdown open AND `filtered.length === 0` AND query is a valid email shape | `e.preventDefault()`. Treat as manual-email fallback (D-12). Add normalized email to `selectedEmails`. Clear input. Show transient row in chip area with the standard email-only chip + tooltip. |
| Enter | Dropdown open AND `filtered.length === 0` AND query is not email-shaped | `e.preventDefault()`. No-op. Empty-results message stays visible. |
| Escape | Dropdown open | Close dropdown (`open=false`). Keep input focused. Do NOT clear query (admin may want to refine). |
| Escape | Dropdown closed | No-op (don't blur). |
| Backspace | Input value is empty AND `selectedEmails.length > 0` | Remove last chip (`selectedEmails.slice(0, -1)`). Don't open dropdown automatically — admin can still keep typing. |
| Backspace | Input has any text | Native input behavior (delete one char). Filter updates. |
| Tab | Dropdown open | Native tab behavior — close dropdown, move focus to next form control (likely the next picker, the Save button, or the next input). Do NOT trap focus. Do NOT use Tab as "select active row". |
| Click on dropdown row | — | Add row's email to `selectedEmails`. Refocus input. Clear query. Keep dropdown open. |
| Click on chip `[×]` | — | Remove that email from `selectedEmails`. Do NOT change input focus. |

**Email-shape check** (for Enter-as-manual-add): regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` after `.trim().toLowerCase()`. Case-insensitive, no leading/trailing whitespace.

**No focus trap.** This is an inline combobox, not a modal. The user must be able to Tab out to Save Changes naturally.

**No multi-token AND search** in v1. Per Claude's Discretion in CONTEXT.md, default is **case-insensitive substring** against the joined searchable string. Specifically, the filter is `fullName.toLowerCase().includes(q) || email.toLowerCase().includes(q) || (division ?? '').toLowerCase().includes(q)` — exactly what RESEARCH Pattern 1 declares. Multi-token search deferred to v2.

**No debounce.** Substring filter on 30 records is sub-millisecond. Filtering on every keystroke is the right answer.

### Refresh Button + Metadata Strip (D-13 — locked)

**Layout:** A single horizontal row at the top of the Visibility tab, above the Dashboard Admins section. Two buttons inline, with metadata text immediately below.

```tsx
<div className="space-y-2 mb-4">
  <div className="flex items-center gap-3">
    <button
      type="button"
      onClick={handleRefresh}
      disabled={busy}
      className="flex items-center gap-2 px-4 py-2 bg-gray-800 hover:bg-gray-700 disabled:opacity-50 text-gray-300 hover:text-white text-sm font-medium rounded-lg border border-gray-700 transition-colors"
    >
      <RefreshCw size={14} className={busy ? 'animate-spin' : ''} />
      {busy ? 'Refreshing…' : 'Refresh Prime Users'}
    </button>
    <button
      type="button"
      onClick={handleSave}
      disabled={!dirty || saving}
      className="flex items-center gap-2 px-4 py-2 bg-red-600 hover:bg-red-700 disabled:opacity-50 text-white text-sm font-medium rounded-lg transition-colors"
    >
      <Save size={14} />
      Save Changes
    </button>
  </div>
  {refreshStatus.kind === 'ok' && (
    <p className="text-xs text-gray-500">
      Refreshed {refreshStatus.userCount} users in {(refreshStatus.durationMs / 1000).toFixed(1)}s · cached {formatRelative(refreshStatus.cachedAt)}
    </p>
  )}
  {refreshStatus.kind === 'idle' && lastSuccessAt && (
    <p className="text-xs text-gray-500">
      Cache last refreshed {formatRelative(lastSuccessAt)}
    </p>
  )}
  {refreshStatus.kind === 'err' && (
    <p className="text-xs text-amber-400">
      Prime unreachable — {refreshStatus.lastSuccessAt ? `using cache from ${formatRelative(refreshStatus.lastSuccessAt)}` : 'no cache available'}
    </p>
  )}
</div>
```

**Copy templates (locked):**

| Status | Copy |
|--------|------|
| Idle, no prior refresh in session, cache exists | `Cache last refreshed 5 days ago` |
| Idle, no cache | `No Prime user cache yet — click Refresh to load.` |
| Refreshing | Button label: `Refreshing…` (replaces `Refresh Prime Users`). No metadata line. |
| Success | `Refreshed 28 users in 1.2s · cached just now` |
| Failure with cache | `Prime unreachable — using cache from 2 days ago` (text-amber-400) |
| Failure without cache | `Prime unreachable — no cache available` (text-amber-400) |

**Relative-time format (`formatRelative()` — locked unit thresholds):**

| Elapsed | Output |
|---------|--------|
| < 60 seconds | `just now` |
| < 60 minutes | `{n}m ago` (e.g. `5m ago`) |
| < 24 hours | `{n}h ago` (e.g. `3h ago`) |
| < 7 days | `{n} days ago` (e.g. `5 days ago`) — pluralize: `1 day ago` for n=1 |
| < 30 days | `{n} weeks ago` |
| ≥ 30 days | `{n} months ago` |

This is the single canonical `formatRelative()` helper. RESEARCH §"Standard Stack" and `lib/format-relative.ts` reference confirm this is a new utility.

**No "ago" suffix when displaying "just now"** — it's already complete.

**No live tick.** The metadata line is set at fetch resolution and at component mount. Does not auto-update. If the admin sits on the page for 10 minutes, the line still says "5 days ago"; it does not crawl to "5 days, 10 minutes ago". Refreshing the page (or clicking Refresh) updates the value. This matches RESEARCH "Don't Hand-Roll" — no `setInterval` for clock updates.

**Copywriting rules (locked):**

- **Sentence case throughout.** Buttons, headings, copy. Title Case is reserved for the existing tab buttons ("Visibility", "Audit Log") which are already Title Case — Phase 3 doesn't touch those.
- **No exclamation marks.** No "Saved!" no "Done!" — neutral confirmation copy only.
- **No emoji.** Never. (Per project CLAUDE.md: "Only use emojis if the user explicitly requests it.")
- **Periods at end of complete sentences.** Single phrases without verbs (button labels, headings) have no period.
- **Numbers always rendered as Arabic numerals** in metadata. `Refreshed 28 users in 1.2s` — never "twenty-eight".
- **Pluralization:** `1 day ago` for n=1, `2 days ago` for n=2+. Same for `user` / `users`, `month` / `months`, `week` / `weeks`. The `formatRelative()` helper handles this.
- **Address the admin in second person where possible**, but never use "you" gratuitously. Empty states explain state, not blame ("No members" not "You haven't added members").
- **Destructive actions copy:** Removing a chip or group member has NO confirmation dialog (matches existing UX — chip remove is `[×]` click and the change is local-state until Save). Per D-21 the only persisted commit is the existing Save Changes button. The implicit "saving will undo if you reload" is the affordance. No "Are you sure?" copy in this phase.

## Accessibility (consolidated)

| Concern | Treatment |
|---------|-----------|
| Combobox ARIA | Locked — see ARIA Contract above. `combobox` + `listbox` + `option` per W3C APG. |
