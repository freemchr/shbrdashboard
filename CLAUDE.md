# SHBR Dashboard — Claude Code Instructions

Next.js 14 dashboard for SHBR Group surfacing Prime ERP data to internal staff. Production product, not a sandbox.

## Planning & Workflow

This project uses GSD (Get Shit Done) for structured execution. Canonical planning docs live in `.planning/`:

- **`.planning/PROJECT.md`** — project context, core value, current milestone, key decisions. Read first when joining a session.
- **`.planning/REQUIREMENTS.md`** — milestone requirements with REQ-IDs (DIR-*, SESSION-*, ADMIN-*, DISPLAY-*).
- **`.planning/ROADMAP.md`** — phase structure with success criteria and traceability.
- **`.planning/STATE.md`** — current position, open questions, deferred items.
- **`.planning/codebase/*.md`** — architecture / stack / structure / conventions / concerns analysis (generated by `/gsd-map-codebase`).
- **`.planning/config.json`** — workflow settings (mode: yolo, granularity: standard, model: balanced, research + plan-check + verifier all on).

Common commands:
- `/gsd-progress` — where are we, what's next
- `/gsd-plan-phase [N]` — detailed plan for a phase (spawns research + plan-check)
- `/gsd-execute-phase` — run all plans in current phase
- `/gsd-verify-work` — conversational UAT against requirements

## Tech Stack (for reference)

- **Framework**: Next.js 14.2.35 (App Router) + React 18 + TypeScript 5
- **Styling**: Tailwind CSS 3.4 (dark theme, brand red `#DC2626`)
- **Deployment**: Vercel; `@vercel/blob` for config/audit/cache persistence
- **Auth**: iron-session cookie (8h TTL) over Prime OAuth password grant
- **Data**: Prime ERP v2 API via `lib/prime-auth.ts` (token cache + paginated fetch with 1.1s throttle)
- **AI**: OpenAI GPT-4o for report-assist features
- **Testing**: None installed (no Jest/Vitest). Test strategy decided per milestone.

## Prime API Constraints

- OAuth password grant at `/oauth/token`; tokens cached in-memory, refresh 30s before expiry
- Rate limits: **60 req/min**, **5,000 req/rolling 24h**, **5 concurrent** — shared across the entire app
- No webhooks — all Prime-sourced data is poll-only
- Response format is JSON:API v1.0 (type/id/attributes/relationships/meta)
- ~47 of Prime's ~55 resources are currently unused (see conversation notes or ask for a Prime capability briefing)

## Ways of Working

- **Brownfield-aware**: the dashboard has been iterated on for multiple quarters. Prefer editing existing files and patterns over introducing parallel structures.
- **Admin fallback email**: `lib/page-visibility.ts` currently hardcodes `chris.freeman@techgurus.com.au` as the admin fallback. Known issue; do not propagate.
- **Existing VisibilityConfig blob schema is load-bearing** — production blobs are in use; schema changes require migration plans.
- **Cron jobs** are live (see `vercel.json`): client/location analytics refresh Fridays 8am AEDT, geocode daily 3pm, timeline daily 8pm, SLA predict daily 7pm, BOM warnings Mondays 1am.
- **Cache before you call Prime** — the 5,000/day budget is shared. Any new Prime integration must declare its caching strategy.

## Code Conventions

- Pages are client components (`'use client'`) that fetch via dedicated `/api/prime/*` routes
- New pages must be registered in `lib/page-visibility.ts` `ALL_PAGES` AND in the Sidebar nav (known drift risk; see v2 NAV-DRIFT requirement)
- Component naming: PascalCase in `components/ui/` and `components/charts/`
- Utility files: kebab-case in `lib/`
- API routes: `app/api/[domain]/[endpoint]/route.ts` pattern; Prime-backed routes call `primeGet()` / `primeGetAllPages()`
- Error handling: log internally with full detail, return generic user-facing messages
